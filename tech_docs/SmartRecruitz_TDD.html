Scope & Objectives

What we're building and what we're not building in this phase.

Objective

Build 5 AI-powered agents that process candidates from resume upload to verified talent pool entry. Each agent runs independently, called sequentially through a chatbot-style frontend interface.

In Scope — 5 Agents

#AgentPurpose
1Resume Parser & Skill NormalizationParse resume, extract structured data, normalize skills to taxonomy
2Duplicate DetectionDetect existing profiles using DB + AI semantic matching
3ID VerificationVerify identity documents against claimed data
4Interview Question GenerationGenerate personalized interview questions from candidate profile
5Interview Scoring & ReadinessScore interview transcript, assign readiness score

Tech Stack

Technologies and frameworks used across the platform.

LayerTechnologyRationale
Backend APIFastAPI (Python 3.12)Async, auto-docs, type-safe, fast
AI ModelClaude Sonnet (Anthropic SDK)All 5 agents — text + document/image analysis
DatabasePostgreSQLJSONB for flexible data, relational for structured
ORMSQLAlchemy (async)Type-safe models, migrations via Alembic
Task QueueARQ + RedisAsync background processing (AI calls take 5-15 sec)
ValidationPydanticInput/output schema validation
File StorageLocal (dev) / S3 (prod)Resume, ID document storage
ContainersDocker + ComposePostgres, Redis, app, worker in one command
Why no LangChain / LangGraph? All agents are single-call to Claude — no chaining, no branching, no AI-controlled flow. The pipeline is linear and code-controlled. Anthropic SDK directly is simpler, faster, and easier to debug.

Candidate Pipeline Flow

End-to-end journey from resume upload to talent pool entry.

IN
Candidate Registration
Chatbot collects name, email, phone — saved to database. No AI needed.
↓
A1
Agent 1 — Resume Parser & Skill Normalization
Candidate uploads resume. AI reads PDF/DOCX/Image directly, extracts all fields, normalizes skills to standard taxonomy, identifies domains, infers implied skills. Also does quick duplicate pre-check (email/phone exact match).
Claude Sonnet (with file attached)
→ Saved to candidates_staging (status: PENDING)
↓
A2
Agent 2 — Duplicate Detection
Step 1 (DB — free): Query candidates_main for potential matches using email, phone, fuzzy name, company+dates. Narrows thousands → 5-10 candidates.
Step 2 (AI — one call): Claude compares new candidate against 5-10 matches. Semantic name matching (Bob=Robert), timeline validation, confidence scoring.
DB Query + Claude Sonnet
→ UNIQUE: Move to candidates_main (status: VERIFIED)
→ DUPLICATE (>75%): Merge or reject
→ UNCERTAIN (50-75%): Flag for HR review
↓
A3
Agent 3 — ID Verification
Candidate uploads ID document (Aadhaar/PAN/Passport). AI reads image, extracts text via OCR, checks for tampering indicators, compares extracted data against claimed registration data.
Claude Sonnet (with image attached)
→ VERIFIED: status → IDENTITY_VERIFIED
→ FAILED: status → VERIFICATION_FAILED
Note: No connection to government databases — internal consistency check only.
↓
A4
Agent 4 — Interview Question Generation
AI analyzes candidate's specific experience, skills, and target role. Generates 10-15 personalized questions that target claimed expertise. Each question is hashed and checked against history to prevent repetition.
Claude Sonnet
→ Questions + expected answers stored in DB. Interviewer guide generated.
↓
VI
Video Interview (External — Interview as Service Module)
Candidate records video answers. External service handles proctoring, face matching, liveness detection. SmartRecruitz receives the transcript + genuineness flags.
Separate Module — Not in this build
↓
A5
Agent 5 — Interview Scoring & Readiness
AI evaluates interview transcript against expected answers from Agent 4. Scores each answer (CORRECT/PARTIAL/INCORRECT), assesses technical knowledge and communication, generates overall readiness score.
Claude Sonnet
→ Score ≥ 60%: status → TALENT_POOL ✅
→ Score < 60%: status → INTERVIEW_FAILED (retry after 30 days)
↓
✓
Verified Talent Pool
Candidate is now verified, interviewed, and scored. Visible to hiring managers for job matching (Agent 6 — future phase).

Agent Breakdown

Detailed input, output, and AI model for each agent.

1
Resume Parser & Skill Normalization
Claude Sonnet — with PDF/DOCX/Image file attached

Parses unstructured resumes into structured candidate profiles. Normalizes skills to standard taxonomy. Identifies domains per experience entry and calculates domain-wise experience.

Input
  • Resume file (PDF, DOCX, or Image)
  • Candidate reference ID
  • Extraction config flags
Output
  • Personal info (name, email, phone, title)
  • Experience entries with domain classification
  • Education, certifications, projects
  • Normalized skills with proficiency levels
  • Implied skills with confidence scores
  • Domain-wise experience breakdown
  • Overall confidence score (0-1)
Quick Duplicate Pre-Check: After parsing, does an exact email/phone match against candidates_main (DB query, no AI). Flags obvious duplicates immediately — saves an AI call.
2
Duplicate Detection
DB Query (free) + Claude Sonnet (one call)

Two-step process to find duplicates without sending all records to AI.

Step 1 — Database Filtering (Free, Milliseconds)

QueryWhat It Finds
Email exact matchSame email in candidates_main
Phone exact match (normalized)Same phone after removing +, -, spaces
Fuzzy name match (pg_trgm)Similar names (score > 80%)
Company + dates overlapSame employer in same period

Narrows thousands of records → 5-10 potential matches.

Step 2 — AI Verification (One Claude Call)

Sent to AI
  • New candidate profile
  • 5-10 potential matches from DB
  • Confidence threshold (default: 0.75)
AI Returns
  • Duplicate yes/no + confidence %
  • Match evidence (name, phone, experience)
  • Semantic analysis (Bob = Robert)
  • Recommendation: MERGE / REJECT / REVIEW
Name alone is never enough. AI only flags duplicates when name + at least one other signal match (phone, email, company+dates, education+year).
3
ID Verification
Claude Sonnet — with ID document image attached

Verifies candidate identity by reading uploaded ID documents and comparing against claimed registration data.

Input
  • ID document image (Aadhaar / PAN / Passport)
  • Claimed data from registration (name, DOB, gender)
Output
  • Extracted data from document (OCR)
  • Field-by-field match scores (name, DOB, etc.)
  • Document authenticity check (tampering indicators)
  • Overall verification status + confidence
Important: We do NOT connect to Aadhaar/PAN government databases. We verify internal consistency between what the document says and what the candidate claimed during registration.
4
Interview Question Generation
Claude Sonnet

Generates personalized L1 interview questions based on candidate's actual experience and target role.

Input
  • Candidate profile + skill profile (from Agent 1)
  • Experience highlights with responsibilities
  • Target role + generic role template
  • Question config (count, difficulty, categories)
Output
  • 10-15 personalized questions
  • Category per question (Technical / Behavioral / Experience)
  • Expected answer points for each question
  • Follow-up questions
  • Interviewer guide (focus areas, red flags)

Question Non-Repetition

MethodHow It Works
Question HashingEach question is hashed and stored — prevents exact duplicates
Semantic CheckAI compares against recent history — prevents paraphrased duplicates
Experience TargetingQuestions target unique experience — naturally different per candidate
Candidate HistoryIf re-applying, system knows previous questions — never repeats
5
Interview Scoring & Readiness
Claude Sonnet

Analyzes interview transcript and scores each answer against the expected answer points generated by Agent 4.

Input
  • Interview transcript (from video service)
  • Questions + expected answers (from Agent 4)
  • Evaluation criteria
Output
  • Overall score (0-100)
  • Pass/Fail status (threshold: 60%)
  • Per-question score + quality assessment
  • Category breakdown (technical, communication, etc.)
  • Recommendation (PROCEED / REJECT / REVIEW)
  • Readiness level

Thin HTTP routes. Receives requests, validates input, returns task_id or results. No business logic.

Background Processing Flow

1
API receives request
Creates a background_task record (status: QUEUED) and returns task_id to frontend immediately.
↓
2
ARQ Worker picks up the job from Redis
Worker calls the appropriate Agent → Agent calls Claude API → gets response.
↓
3
Worker saves result to database
Updates background_task (status: COMPLETED). Frontend polls GET /tasks/{id} and gets result.

Shared Infrastructure

ComponentLocationPurpose
Claude Clientapp/core/claude_client.pyWraps Anthropic SDK. send_text() for text agents, send_vision() for file/image agents. Retry logic, token tracking.
File Storageapp/core/file_storage.pyUpload/download files. Local disk (dev) or S3 (prod). get_base64() for sending to Claude Vision.
Task Managerapp/core/background_tasks.pyEnqueues jobs to Redis via ARQ. Creates DB tracking record. Returns task_id.
Base Agentapp/agents/base.pyAbstract base class. All 5 agents extend this. Loads prompts from text files.

Database Schema — 11 Tables

PostgreSQL database design for the candidate pipeline.

Table Relationships

candidates_staging (temporary, raw JSONB from Agent 1) │ │ Agent 2 moves unique candidates ▼ candidates_main (verified, normalized) ├── candidate_skills ──→ skill_taxonomy ├── candidate_experiences ├── candidate_educations ├── verifications (Agent 3) └── interviews ──→ interview_questions (Agent 4 + 5) skill_taxonomy (master skill list) uploaded_documents (resumes, ID docs) background_tasks (async job tracking)
Design Decision: candidates_staging stores raw data as JSONB (not normalized) because it may be rejected or merged. Only when moved to candidates_main does data get normalized into child tables.
